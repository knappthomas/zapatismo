# Implementation Plan – 10-1 – Default Shoe for Strava Sync

## 0. Scope Check
- Requirement sheet present and reviewed: ✅
- Fits Zapatismo architecture constraints: ✅
- In-Scope summary (bullets):
  - One default shoe per user; set/clear on shoe edit page; overview shows default indicator.
  - Strava sync assigns newly created workouts to default shoe when set; otherwise no assignment.
  - Default stored on shoe; deletion of shoe removes default; idempotent re-sync does not change existing workout shoe.
  - Backend unit and integration tests for default-shoe and sync-assignment behaviour.
- Out-of-Scope summary (bullets):
  - Bulk reassignment of existing workouts; multiple defaults or default-per-activity-type; Strava gear mapping; automatic/scheduled sync; OAuth/connection flow changes.
- Key assumptions (only if explicitly stated in requirements):
  - Ticket 10 (Strava sync) is in place; shoes and workouts are per user; sync uses existing idempotent import; one default per user is sufficient.
- UNRESOLVED items blocking implementation (if any): None.

## 3. Backend Plan (NestJS)
### 3.1 Modules / Components to touch
- Module(s): ShoesModule (unchanged exports); StravaModule (add import ShoesModule).
- Controller(s): ShoesController – no new endpoints; PATCH `shoes/:id` already used for update.
- Service(s): ShoesService (update logic, findDefaultShoeId, toResponse); StravaService (sync: resolve default shoe, pass shoeId to createByExternalId).
- Repository/Prisma access layer: Prisma only (Shoe model gain `isDefault`).

### 3.2 REST Endpoints
| HTTP | Path | Purpose | Request DTO | Response DTO | AuthN/AuthZ | Errors |
|------|------|---------|-------------|--------------|-------------|--------|
| (unchanged) | GET /api/shoes | List shoes | — | ShoeResponseDto[] (add isDefault) | JWT, USER | — |
| (unchanged) | GET /api/shoes/:id | Get one shoe | — | ShoeResponseDto (add isDefault) | JWT, USER | 404 |
| (unchanged) | PATCH /api/shoes/:id | Update shoe (incl. set/clear default) | UpdateShoeDto (add optional isDefault) | ShoeResponseDto | JWT, USER | 400, 404 |
| (unchanged) | POST /api/strava/sync | Sync (internal: uses default shoe when creating workouts) | SyncRequestDto | SyncResponseDto | JWT, USER | 400, etc. |

Rules: Stateless REST only. No business logic in controllers beyond orchestration/validation. Explicit DTOs; no Prisma entities exposed.

### 3.3 Validation & Error Handling
- Input validation approach: `UpdateShoeDto` – optional `isDefault` boolean via `@IsOptional()` and `@IsBoolean()` (class-validator). Shoe ownership enforced in service (findFirst by id + userId).
- Domain validation approach: When setting `isDefault: true`, clear any other shoe’s default for same user in same update flow (e.g. transactional or two steps: clear others then set this). Setting `isDefault: false` only updates that shoe.
- Error mapping (HTTP status + payload shape): 404 when shoe not found or not owned; 400 for validation (existing ValidationPipe). No new error shapes.
- Logging/Auditing considerations: Optional log when default is set or cleared (no secrets).

## 4. Data / Prisma Plan
### 4.1 Prisma schema changes
- Models to add/modify: **Shoe** – add `isDefault Boolean @default(false) @map("is_default")`. No new model; no separate “default” entity.
- Relations/indices constraints (conceptual): No unique DB constraint for “at most one default per user” (MySQL partial unique index not used); enforce in application in ShoesService.update.
- Backfill / defaults (if required): Existing rows get `isDefault: false` via schema default; no backfill script.

### 4.2 Migration steps
- Migration name suggestion: `add_shoe_is_default` (or similar).
- Steps to generate/apply migration: From repo root, `npx prisma migrate dev --name add_shoe_is_default`; then `npx prisma generate`. Commit migration files.
- Seed/test data impact (if relevant): Seed does not set default. Test migrations (e.g. shoes for thomas) may optionally set one shoe as default for integration tests; optional.

Rules: Prisma Migrate only; no manual SQL.

## 10. Execution Sequence
1. **Prisma:** Edit `prisma/schema.prisma` (add `isDefault` to Shoe). Run `npx prisma migrate dev --name add_shoe_is_default`, then `npx prisma generate`. Commit migration.
2. **Backend – Shoes:** Add `isDefault` to `ShoeResponseDto` and `UpdateShoeDto`. In `ShoesService`: implement findDefaultShoeId(userId); in update(), when dto.isDefault === true, clear other shoes’ isDefault for userId then set this shoe isDefault true; when dto.isDefault === false, set this shoe isDefault false. Extend toResponse to include isDefault. Ensure create/remove unchanged (remove already blocks when workouts linked).
3. **Backend – Strava:** Import ShoesModule in StravaModule. In StravaService.sync(), before loop, call findDefaultShoeId(userId); in createByExternalId call, pass shoeId: defaultShoeId ?? undefined (or omit when null). No change to idempotency (createByExternalId already returns without updating existing).
4. **OpenAPI:** Confirm DTOs have @ApiProperty; run backend, check `/api/docs`.
5. **Frontend:** Add `isDefault` to `Shoe` and `UpdateShoePayload`. In ShoeFormComponent (edit mode only), add checkbox bound to form and include in update payload. In ShoesGridPartComponent and overview list, show default indicator when shoe.isDefault.
6. **Unit tests:** Add/update shoes.service.spec.ts and strava.service.spec.ts as in section 8.
7. **Integration tests:** Add/update shoes.integration-spec.ts (set default, change default) and strava.integration-spec.ts (sync with default, without default, re-sync).
8. **E2E (if applicable):** Extend shoes E2E for set/clear default and overview indicator.
9. **Run:** Unit tests, then integration tests (MySQL up, DATABASE_URL), then manual/E2E smoke.
