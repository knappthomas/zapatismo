# Implementation Plan – 10-2 – Sync UX & Display Optimizations

## 0. Scope

- **In scope**
  1. When user clicks “Sync Strava” and the date popup opens: check if user has a default shoe; if not, show a warning in the modal (no block – user can still sync).
  2. Frontend only: display all kilometer values with exactly 2 decimal places (e.g. 10,12 km); no DB or API change.
  3. Frontend only: hide every display of steps / step-count (Strava does not deliver steps).

- **Out of scope**
  - Backend or API changes for (2) and (3).
  - Changing workout form steps field behavior (e.g. optional vs required) unless we hide the field; see 3.2.

- **Assumptions**
  - Default shoe is already implemented (10-1); shoes list includes `isDefault`.
  - Locale for number formatting may vary (comma vs point); Angular `number` pipe with `'1.2-2'` gives exactly 2 fraction digits in locale format.

## 2. Work Breakdown

### WP-1: Default shoe warning in Sync Strava modal

- **Goal:** When the Sync Strava modal is opened (date step), if the user has no default shoe, show a warning message. Sync remains allowed.
- **Location:** `apps/frontend/src/app/features/workouts/workouts-overview.component.ts`
- **Implementation:**
  - Inject `ShoesService`.
  - In `openSyncModal()`: after opening the modal and (in parallel or after) setting `syncFromDate` via `getLastSync()`, call `ShoesService.getList()` and set a signal e.g. `hasDefaultShoe` from `shoes.some(s => s.isDefault)`.
  - In the modal template: if `hasDefaultShoe() === false`, show a warning block (e.g. `alert alert-warning`) with text like: “You don’t have a default shoe set. Synced workouts will not be assigned to a shoe. You can set a default in Shoes.”
  - Use `data-cy="sync-no-default-shoe-warning"` for E2E.
- **Edge cases:** Loading state (shoes not yet loaded): can show warning only when list has been loaded and no default exists; or show “Checking…” then warning/ok. Prefer: load shoes once modal opens, then show warning when `shoes.length > 0 && !shoes.some(s => s.isDefault)` or when `shoes.length === 0` (no shoes → no default).

**Clarification:** “No default” = either no shoes at all, or no shoe with `isDefault === true`. Warning in both cases.

---

### WP-2: Kilometer display – exactly 2 decimal places (frontend only)

- **Goal:** Every place that displays km (distance or total distance) shows exactly 2 decimal places (e.g. 10,12 or 10.12 depending on locale). Display only; no DB/API change.
- **Locations and changes:**

| File | Current | Change |
|------|---------|--------|
| `workouts-list-part.component.ts` | `{{ workout.distanceKm }}` | `{{ workout.distanceKm \| number : '1.2-2' }}` (and ensure `DecimalPipe` or `Number`/locale pipe is imported if not already; component already has `DatePipe`, add `DecimalPipe` if needed or use `number` from `@angular/common`). |
| `shoes-grid-part.component.ts` | `{{ shoe.totalDistanceKm \| number : '1.1-1' }}` and `'1.2-2'` in aria-label | Use `'1.2-2'` for both visible text and aria-label so exactly 2 decimals everywhere. |

- **Other fronts:** Check dashboard and any other components that display distance; they use `WorkoutsListPartComponent` and `ShoesGridPartComponent`, so fixing those two covers all list/grid displays. Workout form: distance **input** can keep `step="0.01"`; no need to change the stored value, only displayed values in list/grid/summaries.
- **Format:** Angular `number` pipe with `'1.2-2'` = min 2, max 2 fraction digits. Locale (e.g. de: “10,12”, en: “10.12”) is handled by the pipe.

### WP-3: Hide steps / step-count in the frontend

- **Goal:** No visible display of steps or step-count anywhere in the UI (because Strava does not deliver steps).
- **Locations and changes:**

| File | Current | Change |
|------|---------|--------|
| `workouts-list-part.component.ts` | Table column “Steps” and `{{ workout.steps }}` | Remove the `<th>Steps</th>` and the `<td>{{ workout.steps }}</td>` column. |
| `shoes-grid-part.component.ts` | Line “Steps: {{ shoe.totalSteps }}” | Remove the whole `<p>` that contains `shoe-total-steps` (or remove only the steps line if other content shares the block). Current: single line `<p><span data-cy="shoe-total-steps">Steps: {{ shoe.totalSteps }}</span></p>` → remove that `<p>`. |
| `shoes-overview.component.ts` | List view table | No “Steps” column in list view currently; only grid shows steps via `shoes-grid-part`. So no change in overview table. |
| `workout-form.component.ts` | Steps label + input (required) | **Option A:** Hide the Steps field in the form (e.g. `*ngIf="false"` or remove block). Then form must submit a value for `steps` (backend may require it): use a default (e.g. 0) when creating/updating from form. **Option B:** Keep the Steps field for manual “Add workout” entry but hide it everywhere in read-only views. Plan chooses **Option B** (hide only read-only displays) unless product decision is to remove steps from form too; then backend must allow steps to be optional or default 0. |

**Decision for plan:** Hide steps only in:
- Workouts list (workouts-list-part): remove Steps column.
- Shoes grid (shoes-grid-part): remove “Steps: {{ shoe.totalSteps }}” line.

Keep the Steps field in the workout form for manual entry; backend already has steps. If product later decides to remove steps from form (e.g. make optional or default 0), that can be a follow-up.

- **E2E / fixtures:** Tests that assert on steps (e.g. “Steps: 12500” in shoes grid, or Steps column in workouts list) must be updated: remove or rewrite assertions that reference steps. See WP-5.

## 3. Testing Plan

### 3.1 Default shoe warning (WP-1)

- **E2E (Cypress), fixtures**
  - **Sync modal shows warning when user has no default shoe**
    - Intercept: `GET /api/strava/last-sync` (or equivalent) and `GET /api/shoes` with fixture `shoes/loaded.json` (no default: all `isDefault: false`) or `shoes/empty.json`.
    - Visit `/workouts`, click “Sync Strava”, wait for modal.
    - Assert: element `[data-cy="sync-no-default-shoe-warning"]` is visible and contains text about no default shoe.
  - **Sync modal shows no warning when user has default shoe**
    - Intercept `GET /api/shoes` with `shoes/loaded-with-default.json`.
    - Visit `/workouts`, click “Sync Strava”.
    - Assert: `[data-cy="sync-no-default-shoe-warning"]` does not exist or is not visible.
  - **Sync still works when no default (optional)**  
    - Can be covered in same flow: confirm sync button is enabled and (if stubbing POST sync) that sync can be triggered.

- **Page object**
  - In `WorkoutsOverviewPO`: add getter for sync modal (e.g. `syncModal`) and for warning (e.g. `syncNoDefaultShoeWarning`). Add getter for “Sync Strava” button if not present (e.g. `syncStravaButton`).

### 3.2 Kilometer display – 2 decimals (WP-2)

- **E2E**
  - Workouts list: intercept `GET /api/workouts` with fixture that has a workout with `distanceKm` with more than 2 decimals (e.g. 10.1234). Visit workouts list, assert the cell shows exactly 2 decimals (e.g. “10,12” or “10.12” depending on locale).
  - Shoes grid: intercept `GET /api/shoes` with fixture with `totalDistanceKm` like 120.567; assert displayed text shows 2 decimals (e.g. “120,57” or “120.57”).

- **Note:** Locale in Cypress may be en-US so expect “10.12”. Prefer asserting with a regex or “contains” for the integer part and 2 decimal places, to avoid locale brittleness.

### 3.3 Hide steps (WP-3)

- **E2E**
  - **Workouts list:** With workouts fixture that has `steps`, visit `/workouts` (or dashboard workouts section); assert that there is no column header “Steps” and no cell showing step count. Update `workouts.cy.ts`: remove any step-related assertions; adjust “overview shows workouts when list returns data” if it asserted on steps.
  - **Shoes grid:** Remove or change test “grid shows total steps and distance progress bar per shoe”: drop assertion on `shoeTotalSteps` (e.g. “12500”); keep assertion on distance progress bar. Optionally add explicit test “grid does not show steps” (assert `[data-cy="shoe-total-steps"]` does not exist).
  - **Workout form (if kept):** Existing smoke/fixture tests that fill steps can remain (form still has steps for manual add). No new test required for “steps hidden” on form if we keep the field.

- **Page objects**
  - `ShoesOverviewPO`: remove or repurpose `shoeTotalSteps` if it’s only used for steps; if used elsewhere, keep but don’t assert steps content in 10-2 tests.
  - `WorkoutFormPO`: keep `stepsInput` for manual-add flows.
